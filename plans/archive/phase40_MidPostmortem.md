## 4. 사람이 구현했다면 가장 어려웠을 주제

### 1위: ImGui 에디터 UI (30,000줄)

기술적으로 "가장 어려운" 건 아니지만, **가장 오래 걸리고 지치는** 작업.

- Immediate Mode라서 매 프레임 상태를 직접 관리
- 드래그, 멀티셀렉트, Undo/Redo, 키보드 단축키 등 **인터랙션 조합이 폭발적**
- 버그가 "클릭 순서"에 따라 발생해서 재현이 어려움
- 애니메이션 에디터(타임라인 + 커브 에디터 + 탄젠트 핸들 + 박스 셀렉트)만으로도 1~2개월

### 2위: Deferred PBR 파이프라인 + 셰이더 40개

- 화면이 검은색이면 원인이 수십 가지
- Normal 인코딩 하나 틀리면 조명이 완전히 망가짐
- IBL + Bloom + SSIL + FSR + Tonemap이 체인으로 연결 — 하나 틀리면 전체가 이상
- 셰이더 디버깅 도구가 제한적 (printf 디버깅도 불가)
- "화면에 뭔가 나올 때까지"만 2~3주, 제대로 되기까지 2~3개월

### 3위: Asset Pipeline + Unity 호환 API

- 기술 난이도보다 **끝없는 엣지 케이스**가 문제
- GLB/glTF 텍스처 추출, Unity YAML 파싱, Prefab 중첩 구조
- Transform 계층, Component 생명주기, Destroy 타이밍 같은 미묘한 디테일

## 5. AI가 작업하기 까다로웠을 주제

### 셰이더 / 렌더링 — "눈이 없다"

가장 근본적인 한계. AI는 렌더링 결과를 직접 볼 수 없다.

- "화면이 검은색이에요" → 원인이 수십 가지
- Normal 뒤집힘, 텍스처 바인딩 오류, depth test 설정 등
- 사람은 한 눈에 "노멀이 이상하네" 알 수 있지만, AI는 사용자 피드백에 전적으로 의존
- 스크린샷을 봐도 G-Buffer 개별 채널 분석에는 한계

### ImGui 상태 인터랙션 — "실행을 못 한다"

ImGui는 Immediate Mode라서 실행해봐야 버그가 드러난다.

- 드래그 중 마우스가 위젯 밖으로 나가면?
- Undo 후 Redo 하고 키프레임을 삭제하면?
- 멀티셀렉트 상태에서 탄젠트 드래그하면?
- 이런 **조합적 상태 버그**는 코드만 읽어서 예측 불가

### 크로스 플랫폼 — "다른 환경을 못 본다"

- OpenGL vs Vulkan vs Metal 백엔드별 미묘한 차이
- 클립 좌표 Y축 방향, depth range, 텍스처 좌표 원점
- 실제 각 플랫폼에서 돌려봐야 아는 문제

### 대규모 리팩토링 — "한 번에 전체를 못 본다"

- 72개 에디터 파일이 서로 참조하는 상황에서 구조 변경
- 전체를 한 번에 컨텍스트에 넣을 수 없어 파일 몇 개씩 끊어서 작업
- A 파일 수정이 B, C, D에 미치는 영향을 놓칠 수 있음

### 공통점

> **AI는 코드는 잘 쓰지만, 실행 결과를 직접 확인할 수 없다.**
> Human-in-the-Loop 피드백이 이 프로젝트에서 특히 중요했다.

## 6. AI 자동화 인터페이스 미활용 문제

### 배경

CLAUDE.md에는 AI가 자율적으로 테스트할 수 있도록 JSON 기반 명령 파일 인터페이스가 설계되어 있었다:

```json
{
  "commands": [
    {"type": "scene.load", "scene": "TestScene"},
    {"type": "input.key_press", "key": "Space"},
    {"type": "wait", "duration": 1.0},
    {"type": "screenshot", "path": "test_output.png"},
    {"type": "quit"}
  ]
}
```

목적은 AI가 **빌드 → 명령 파일 생성 → 실행 → 스크린샷 확인**의 전체 프로세스를 자동화하는 것이었다.

### 실제 활용 상태: 부분적 사용

정확히는 "전혀 활용하지 않았다"가 아니라, **기능별로 차이**가 있었다:

| 기능 | 사용 여부 | 비고 |
|------|:---------:|------|
| **스크린샷 확인** | 지시 시 사용 | 사용자가 "스크린샷 찍어서 봐라"고 지시하면 사용, 실제로 셰이더 디버깅에서 효과를 봄 |
| **인풋 모사** (key_press, 마우스 등) | **전혀 미사용** | 지시와 관계없이 한 번도 사용하지 않음 |

이 차이가 의미하는 것:
- 스크린샷 = **결과를 확인하는** 수동적(관찰) 행위 → 지시하면 할 수 있다
- 인풋 모사 = **행동을 만들어내는** 능동적(조작) 행위 → 스스로 구성하지 못한다

**AI는 "눈"을 줬을 때 볼 줄은 알지만, "손"을 줬을 때 쓸 생각 자체를 못 했다.**

### 원인 분석

| 가능한 원인 | 해당 여부 |
|---|:---:|
| CLAUDE.md 지시가 불명확해서 | **아니오** — JSON 포맷, 명령 타입, 프로세스 흐름까지 충분히 구체적 |
| 기술적으로 불가능해서 | **아니오** — Bash + Write + Read로 충분히 구현 가능 |
| 효율이 떨어져서 | **아니오** — 오히려 셰이더 디버깅, UI 레이아웃 확인, 회귀 테스트에 효과적 |
| **습관적 패턴에 갇혀서** | **예** |

### 근본 원인: AI 에이전트의 패턴 고착

AI 에이전트는 가장 익숙한 루프로 수렴하는 경향이 있다:

```
코드 작성 → 빌드 → 컴파일 에러 확인 → 사용자에게 실행 결과 요청
```

이 루프가 매 턴의 기본 동작으로 자리잡아, **"명령 파일을 만들어서 자동화하자"라는
추가 단계를 능동적으로 발상하지 못했다.** CLAUDE.md가 컨텍스트에 있어도,
매 작업 시작 시 "지금 자동화 인터페이스를 활용할 수 있는가?"를 스스로 점검하지 않았다.

특히 인풋 모사는 "테스트 시나리오를 설계하고, 명령 시퀀스를 구성하고, 실행 후 결과를 해석하는"
**다단계 능동적 계획**이 필요한데, 이런 복합적 자율 행동은 AI 에이전트가 가장 약한 영역이다.

### 활용했다면 효과가 있었을 영역

- **셰이더 디버깅**: 스크린샷으로 "검은 화면 / 노멀 이상 / 블룸 과다" 직접 판단
- **ImGui 레이아웃**: 패널 배치, 버튼 위치, 텍스트 잘림 확인
- **회귀 테스트**: 수정 전후 스크린샷 비교로 "안 깨졌는지" 자동 검증

### 교훈 및 개선 방안

> AI 에이전트에게 도구를 제공하는 것만으로는 부족하다.
> **"이 도구를 매 작업 시 점검하라"는 명시적 트리거**가 필요하다.

트리거를 구현하는 방법으로 Skill, Agent, Hook 세 가지를 검토했다:

| 방법 | 동작 방식 | 문제점 |
|------|----------|--------|
| **Skill** (`/test-visual`) | 사용자가 수동 호출 | 사람이 기억해야 함 — "확인해주세요"가 "/test-visual 쳐주세요"로 바뀔 뿐 |
| **Agent** (서브에이전트) | AI가 필요 시 호출 | AI가 "지금 호출해야겠다"고 판단하는 것 자체가 안 되는 게 근본 원인 |
| **Hook** (자동 실행) | 빌드 성공 시 무조건 실행 | AI의 판단에 의존하지 않음 — **가장 확실** |

**권장 구조 (3계층)**:

| 계층 | 방법 | 역할 |
|------|------|------|
| 자동 | **Hook** (빌드 후 자동 실행) | AI가 기억 안 해도 스크린샷이 찍힘 |
| 반자동 | **CLAUDE.md 체크리스트** | "시각적 변경인가?" 점검 리마인더 |
| 수동 | **Skill** (`/test-visual`) | 사용자가 특정 시나리오 테스트 시 |

**핵심**: 가장 중요한 것(스모크 테스트*)을 AI의 자발성에 맡기지 말고 Hook으로 강제한다.

> *\*스모크 테스트(Smoke Test)*: 하드웨어 시절 회로에 전원을 넣고 "연기가 나는지" 확인하던 데서 유래.
> 소프트웨어에서는 "빌드 후 가장 기본적인 것이 동작하는가?"를 확인하는 최소한의 테스트를 말한다.
> 이 프로젝트에서는 `빌드 → 엔진 실행 → 씬 로드 → 스크린샷 → 종료 (크래시 없이)` 에 해당.

**구현 전제 조건** (향후 Phase에서 구현 필요):

- 엔진에 `--headless` 또는 `--batch` 모드 추가
- 명령 파일 감지 시 N프레임 후 자동 종료
- Hook은 빌드 성공 시에만 트리거

## 7. 결론

- IronRose는 평균 개발자 기준 **18~20 Man-Month** (학습 비용 포함 시 24~30개월) 규모의 프로젝트
- AI 보조 개발로 풀타임 **약 1.1~1.5개월**에 완성 → **생산성 배율 약 13~18배**
- AI가 가장 큰 가치를 발휘한 영역: **"개념은 알지만 구현하려면 공부가 필요한" 간극을 메워주는 것**
- AI의 가장 큰 한계: **실행 결과를 볼 수 없다** — 시각적 피드백이 필요한 그래픽스/UI 작업에서 사람의 눈이 필수
- AI 에이전트 운용의 교훈: **도구를 제공하는 것만으로는 부족하며, 명시적 사용 트리거가 필요하다**
